# SYRIX — SpecKit (System + Matrix)

هذا المجلد عبارة عن **SpecKit** كامل لمشروع **SYRIX** — منصة إدارية متكاملة لأكثر من 100 موظف (موارد بشرية، عمليات، حسابات، تسويق، مبيعات).

منهجية SpecKit تعتمد على فصل التخطيط عن التنفيذ بحيث تقدر تدي كل ملف لموديل ذكاء اصطناعي رخيص (زي GPT-4o-mini, Haiku, DeepSeek, Qwen-Coder...) بشكل منفصل، وهو ينفذ بدون ما يحتاج يفهم المشروع كله من الأول.

## ترتيب القراءة والتنفيذ

| # | الملف | الغرض | مين يستخدمه |
|---|-------|-------|--------------|
| 1 | `01-constitution.md` | القواعد الثابتة اللي محدش يخالفها (أمان، RBAC، بنية الكود) | أي موديل قبل ما يبدأ أي مهمة |
| 2 | `02-spec.md` | المواصفات الوظيفية الكاملة لكل موديول (بما فيها موديول الحسابات بالتفصيل) | Product/Analyst + الموديل قبل التنفيذ |
| 3 | `03-technical-plan.md` | المعمارية التقنية، التوفيق بين Vercel + Firebase، الأداء، النشر | الموديل المسؤول عن الـ setup والـ DevOps |
| 4 | `04-tasks.md` | مهام منفذة قابلة للتنفيذ خطوة خطوة، كل مهمة فيها Acceptance Criteria | أي موديل تنفيذي (حتى لو رخيص) — ينفذ Task واحدة في المرة |

## طريقة الاستخدام مع موديلات رخيصة

1. اديله `01-constitution.md` + `03-technical-plan.md` كـ System Prompt ثابت في كل محادثة.
2. اديله Task واحدة بس من `04-tasks.md` في كل مرة (مش الملف كله) — الموديلات الرخيصة أداءها بيقل لو الـ context كبير أو المهمة مركبة.
3. بعد كل Task، شغّل الـ Acceptance Criteria بتاعتها كـ test قبل ما تنتقل للي بعدها.
4. لو الموديل غلط، رجّعله فقط رسالة الخطأ + نفس الـ Task — من غير ما تديله كل السياق تاني.

## ملاحظة مهمة عن الـ Stack

طلبك الأصلي كان يجمع بين Firebase وPostgreSQL/MySQL وRedis — دول متعارضين جزئيًا لأن Vercel/Firebase بيئة serverless مش بتشغّل Redis أو Postgres تقليدي بشكل دائم. التوفيق بينهم موجود بالتفصيل في `03-technical-plan.md` (القسم الأول).
